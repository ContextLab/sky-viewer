# Implementation Plan: Star-Stencil PDF Generator (Print Mode)

**Branch**: `002-stencil-template-pdf` | **Date**: 2026-04-27 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/002-stencil-template-pdf/spec.md`

## Summary

Add a **Print Mode** to sky-viewer that turns the rendered sky into a
printable, tileable, paper-and-tape stencil for spray-painting a
room's ceiling, walls, or floor. The user enters Print Mode from the
top bar, sets the observation (date/time/location/direction), draws
the room with windows and doors, picks paper size and surfaces, and
clicks Compute to generate a multi-page PDF. The PDF starts with a
calibrated cover page (four hole-size markers + instructions), then
pages tile every enabled surface — ceiling, walls (with optional
through-Earth wall projections), and even the floor (which projects
the antipodal sky). Per-feature paint/no-paint toggles let the user
treat windows as either painted-over or cut-out, and no-paint
features double as physical registration anchors via dotted cut lines
on overlapping tile pages.

Technical approach (detailed in `research.md`): one new runtime dep
(`jspdf`, ~60 KB gzipped), pure-TS projection geometry that ray-casts
each celestial body against each surface, an SVG-based room editor
for the floor plan + per-wall elevation, hole sizing by 4-bin
magnitude classes, deterministic tile grid with 1/8″ overlap. State
in `localStorage` under a new `skyViewer.printJob` key, separate from
the main-view observation so the print job survives main-view edits.

## Technical Context

**Language/Version**: TypeScript 5.x, ES2020 (matches the existing app).
**Primary Dependencies (new)**: `jspdf` ^2.5 (runtime, ~60 KB gzipped).
**Primary Dependencies (existing, reused)**: `src/astro/transforms.ts`,
`src/astro/stars.ts`, `src/astro/sun-moon.ts`, `src/astro/planets.ts`,
`src/app/observation-store.ts`, the map picker and city-search modules
from the main feature. No changes to those modules; this feature only
adds new modules under `src/print/` and `src/ui/print-mode/`.
**Dev Dependencies (new)**: `pdf-parse` ^1.1 (test-only, for verifying
generated PDFs).
**Storage**: `localStorage` under `skyViewer.printJob`. Unrelated to
the main-view observation key.
**Testing**: Vitest for projection / tile-grid / pre-flight / PDF
builder; `@playwright/test` for e2e Print Mode scenarios.
**Target Platform**: same as parent feature — current + previous major
of Chrome, Firefox, Safari, Edge on Windows / macOS / Linux / iOS /
Android, desktop and mobile.
**Project Type**: extension to an existing single-project static web
app. No backend.
**Performance Goals**: Compute completes in ≤ 30 s for the canonical
12 × 12 ft ceiling-only case on a mid-tier laptop (SC-002). Print
Mode does not affect the main view's render-loop performance.
**Constraints**: client-side only (FR-016); no runtime network calls;
respects all existing constitution payload budgets after this feature
lands; supports paper sizes from 100 × 150 mm to 600 × 900 mm.
**Scale/Scope**: 4 new modules under `src/print/`, ~6 new files under
`src/ui/print-mode/`, 4 new Vitest files, 4 new Playwright e2e files.
Estimated ~1500 lines of new TS plus ~200 lines of new test code.

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-check after Phase 1 design.*

Checked against the five principles in
[../../.specify/memory/constitution.md](../../.specify/memory/constitution.md):

**Gate 1 — Performance envelope fits budget.**
✅ R13 reconciles the payload: existing ~90 KB code + new ~30 KB +
jsPDF ~60 KB = ~180 KB gzipped, under the 200 KB code budget. Data
is unchanged. The compute pipeline is bounded at 30 s for the
canonical case (R10), and Print Mode runs only when the user opens
it — the main view's performance is unaffected.

**Gate 2 — UI latency targets declared.**
✅ Print Mode entry: ≤ 200 ms (it's a DOM overlay, not a re-render).
Room editor interactions: ≤ 100 ms p95 (SVG drag, same render loop
as the existing UI). Compute: ≤ 30 s (SC-002). The 30 s budget is
explicitly outside the 100 ms input-to-render gate because Compute is
a one-shot operation, not a per-frame interaction; the user is
shown a progress indicator and can cancel.

**Gate 3 — Astronomical model named.**
✅ Reuses the existing `src/astro/transforms.ts` (Meeus + IAU
reductions, ≤ 0.01° accuracy vs. astronomy-engine), `stars.ts`,
`sun-moon.ts`, `planets.ts`. Antipodal projection is a pure
alt-flip (R12). No new astronomical models introduced.

**Gate 4 — Baseline (no-WebGL) behaviour declared.**
✅ Print Mode does not use WebGL. The room editor is SVG; the PDF is
generated by jsPDF using only ES2020 numerics. The feature works
identically on browsers without WebGL. This is a strict
*augmentation* — the main view's WebGL2 / Canvas2D fallback split is
unchanged.

**Gate 5 — Spec Kit conventions.**
✅ Branch `002-stencil-template-pdf`. Spec + plan + research +
data-model + contracts + quickstart all in place. Five
clarifications recorded in `spec.md` under `## Clarifications`.
`.specify/feature.json` updated.

**Structure Decision**: Extends the existing single-project layout
without restructuring. Print Mode is a coherent vertical: pure TS
geometry + PDF builder under `src/print/`, UI overlay under
`src/ui/print-mode/`. No changes to `src/astro/`, `src/render/`, or
the existing `src/app/` modules. The split between `print/` (no DOM)
and `ui/print-mode/` (DOM) mirrors the parent feature's `app/` vs.
`ui/` separation, so projection and PDF builders are headlessly
testable in Vitest's `jsdom` env.
